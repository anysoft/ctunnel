# 安全设计

## 威胁模型与身份

ctunnel 假设网络不可信，攻击者可以观察、注入、重放、重排序、截断流量并主动建立 TCP 连接。程序会认证配置的对端，并保护控制通道的机密性和完整性。它不会隐藏端点 IP、流量大小/时序或隧道的存在，也无法保护已经失陷的主机或暴露的私钥文件。

长期 Monocypher EdDSA-BLAKE2b 密钥用于认证握手。这是核心 `crypto_eddsa_*` 构造（Curve25519 配合 BLAKE2b），不是可选 Ed25519/SHA-512 API。客户端固定服务端公钥；服务端通过精确 `client_id` 选择已授权客户端公钥。签名密钥和 X25519 密钥分别生成，绝不相互转换。

规范握手转录包含带版本标签、客户端 ID、双方新鲜随机数、双方临时 X25519 公钥、提议的密码套件掩码和最终所选套件。签名覆盖 BLAKE2b-256 转录哈希，并为服务端和客户端角色使用不同固定上下文。实现会显式序列化各字段，绝不签名 C 结构体。每次重连都会生成新的随机数和临时密钥。

## 密钥派生、AEAD 与 nonce

X25519 生成临时共享秘密，并拒绝全零结果。KDF 使用握手转录哈希作为 salt，通过 keyed BLAKE2b-512 提取，再把计数器和带版本上下文输入 keyed BLAKE2b-512 分块扩展。不同标签分别派生控制通道 C→S/S→C 密钥、控制 nonce 基值、DATA master、工作连接绑定 MAC 密钥、流消息 MAC 密钥和会话 ID。DATA 密钥和 nonce 基值还会把流 ID、连接随机数和方向加入上下文。

控制帧和加密 DATA 记录使用 XChaCha20-Poly1305。24 字节 nonce 为 `directional_nonce_base[16] || sequence_u64_be`。不同通道、流和方向的 nonce 基值不同。序号从 1 开始，接收方只接受下一个准确值，任一端都会在 64 位序号回绕前关闭连接。序列化帧头或记录头作为 AEAD 关联数据。重连会创建全新的密钥和 nonce 空间。

工作套接字携带 32 字节连接 ID（`monotonic_u64 || random[24]`），以及覆盖协议标签、会话 ID 和连接 ID 的 keyed BLAKE2b-256 认证值。64 项滑动序号窗口允许有限 TCP 到达乱序，同时拒绝重复值和所有过旧序号。`START_STREAM` 和就绪消息使用单独派生的 keyed BLAKE2b 密钥，并绑定会话、服务、流、模式和连接随机数。这些认证值不会直接复用 DATA master。

DATA `required` 使用独立认证的记录进行加密。DATA `disabled` 仍会认证流建立过程，但应用字节以明文转发；只有在经过审计的端到端协议（例如 SSH 或正确配置的 TLS）已经保护这些字节时才应使用。

## 随机数与秘密处理

Linux 使用 `getrandom(2)`；只有系统调用不存在或暂时不可用时，才回退到完整读取 `/dev/urandom`。macOS 和 BSD 使用 `arc4random_buf`，Windows 使用 `BCryptGenRandom`。任何失败都会中止操作。支持的定长比较和秘密擦除分别使用 Monocypher 的 `crypto_verify*` 与 `crypto_wipe`。私钥、会话密钥、认证值、nonce 材料和 payload 都不会写入日志。

## 限制与审查状态

服务端授权限制准确的监听地址/端口范围、服务数、流数、等待连接数和客户端数。帧、ID、地址、payload 和缓冲区大小都有上限。认证失败会触发进程内指数退避限速，但仍建议在防火墙层进行限速。

自定义线协议及其 BLAKE2b KDF 组合尚未接受独立外部审计或形式化验证。Monocypher 是经过审查的依赖，但这不等于 ctunnel 协议或集成已被审计。第一期不提供 PKI/吊销、TLS/X.509、AES-GCM、单一已支持套件以外的算法协商、持久化重放数据库、抗流量分析、DDoS 吸收、主机失陷恢复、0-RTT 或流恢复。
